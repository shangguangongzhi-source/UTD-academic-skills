# 输出格式

## 默认输出结构

每次综述生成任务，必须同时交付以下两份文件：

### 交付物 1：HTML 综述报告

文件名：`review-framework.html`

视觉风格与同系列技能保持一致（Noto Sans SC + Noto Serif SC 字体，白色背景卡片布局，hover 阴影，共识/分歧/缺口三色标签）。

HTML 模板见对应 review_type 碎片（hypothesis.md / topic.md / method.md）。

### 交付物 2：综述元数据

在 HTML 文件末尾以独立区块形式输出，或在对话中以文本形式提供：

```
## 综述元数据

### 检测到的轴值
- 综述类型：[hypothesis / topic / method]
- 目标风格：[cn-core / ssci]
- 综述边界：[时间范围] / [学科范围] / [纳入标准]

### 一句话综述论证
[步骤 1 产出的核心论证句]

### 假设/主题映射表
| 假设/主题 | 归入文献数 | 共识文献 | 分歧文献 | 缺口 |
|---|---|---|---|---|
| [假设 1] | N | M | K | [缺口描述] |
| [假设 2] | N | M | K | [缺口描述] |

### 文献覆盖度
- 精读报告文献总数：N
- 已引用文献数：M
- 覆盖率：M/N = X%
- 未引用文献及原因：[列出未引用的文献及原因]

### 引用密度统计
| 模块 | 段落数 | 平均引用/段 | 顶刊占比 |
|---|---|---|---|
| 横向模块 | N | X | X% |
| 纵向模块 | N | X | X% |

### 假设或缺失输入
- [列出缺失的关键输入或需要用户补充的信息]
```

## 缺失输入处理

如果关键输入缺失（如某条假设没有支撑文献、综述边界不清晰），不要编造。采取以下处理：

1. 在综述正文中写占位符：`[需补充：XX方向的实证证据]`
2. 在元数据的"假设或缺失输入"部分详细列出
3. 在对话中明确告知用户需要补充什么

## 篇幅规范

- 默认综述主体：3000-5000 字
- 横向模块：80%-85% 篇幅
- 纵向模块：15%-20% 篇幅
- 具体篇幅可根据目标风格碎片调整
